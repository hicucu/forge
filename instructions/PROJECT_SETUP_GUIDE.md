# 프로젝트 셋업 지침 (PROJECT_SETUP_GUIDE)

> 신규 프로젝트를 처음 생성할 때 Claude Code(또는 AI)가 참조하는 **공통 부트스트랩 지침**.
> 프로젝트 유형 판별 → 버전 정책 → 코드 품질·로깅·디렉토리·보안 공통 기준까지를 다룬다.
> **언어·프레임워크에 무관하게 적용되는 원칙만** 정리하며, 구체적 설정 파일·디렉토리 템플릿·설치 스펙은 **언어별 세부 파일**로 위임한다 (§7 참조).
> 코딩 규칙 자체는 `DEVELOPMENT.md` / `LANGUAGE_GUIDELINES_*.md`, AI 동작은 `AI_BEHAVIOR.md`, 커밋은 `COMMIT_CONVENTION.md` 참조 (이 문서는 **프로젝트 부트스트랩 전용**).

---

## 0. 셋업 진행 순서 (전체 개요)

```
1. 프로젝트 유형 판별 (의사결정 트리) → 언어/프레임워크 확정
   ↓
2. 모노레포 여부 결정
   ↓
3. 언어별 세부 가이드 참조 → 스펙 설치 + 디렉토리 구조 + 설정 파일 구성
   ↓
4. 버전 정책 적용 (최신 안정화·LTS 우선)
   ↓
5. 코드 품질 게이트 구성 (pre-commit hook + CI lint/test)
   ↓
6. 로깅 전략 적용 (구조화 로깅·Correlation ID)
   ↓
7. 보안 기준 적용 (환경변수·의존성 감사)
   ↓
8. 셋업 완료 검증 (실제 실행으로 동작 증명)
```

> **원칙**: 스택이 모호하거나 선택지가 둘 이상이면 추론 전에 사용자에게 먼저 질문한다 (`AI_BEHAVIOR.md` 질문 게이트).

## 1. 프로젝트 유형 판별 흐름

먼저 프로젝트가 **프론트엔드 / 백엔드 / 풀스택**인지, **단일 앱 / 모노레포**인지를 결정한 뒤 언어·프레임워크를 확정한다.

### 1.1 프론트엔드 의사결정 트리

```
SSR / SEO 필요한가?
├─ Yes ─ 풀스택 SSR 필요? ── Yes → SSR 프레임워크 (예: Next.js)
│                          └─ No, 정적 위주 → SSG / 정적 사이트 빌더
└─ No ── 순수 SPA → SPA 번들러 + UI 라이브러리 (예: Vite + React)
         모바일/크로스플랫폼 → 크로스플랫폼 프레임워크
```

| 조건                        | 선택 기준                      | Why                                      |
| --------------------------- | ------------------------------ | ---------------------------------------- |
| SSR/SEO 필요, 풀스택        | **SSR 프레임워크**             | 서버 렌더링·라우트 핸들러로 BE/FE 일체화 |
| 정적 콘텐츠 위주 (블로그)   | **SSG / 정적 사이트 빌더**     | 제로/최소 JS, 콘텐츠 중심 최적           |
| 순수 SPA (내부 툴·대시보드) | **SPA 번들러 + UI 라이브러리** | 빠른 HMR, 가벼운 빌드                    |
| 모바일/크로스플랫폼         | **크로스플랫폼 프레임워크**    | 단일 코드베이스로 다중 플랫폼            |

### 1.2 백엔드 의사결정 트리

언어 선택 기준 (팀 역량·생태계·런타임 제약을 종합 판단):

| 조건 / 선호                     | 언어·런타임 선택 기준                                               |
| ------------------------------- | ------------------------------------------------------------------- |
| 엔터프라이즈·강타입·기존 .NET   | **C# / .NET** (최신 LTS, 레거시 .NET Framework 금지)                |
| JS/TS 생태계 통합·풀스택 동질성 | **Node.js** (DI·모듈 구조 필요 시 프레임워크, 단순 API는 경량 서버) |
| 타입힌트·async·데이터/ML 친화   | **Python** (비동기 API 또는 풀스택 프레임워크)                      |

> 언어 선택 기준이 모호하면 사용자에게 우선순위(팀 숙련도·성능·생태계·배포 환경)를 질문한 뒤 결정한다.

### 1.3 모노레포 여부

| 조건                | 선택            | Why                           |
| ------------------- | --------------- | ----------------------------- |
| 다수 앱/공유 패키지 | **모노레포**    | 의존성 공유 절약, 태스크 캐싱 |
| 단일 앱             | **단일 저장소** | 오버엔지니어링 회피           |

> 모노레포 선택 시 패키지 매니저를 단일 도구로 강제하여 혼용을 차단한다 (구체 도구·설정은 언어별 가이드 참조).

## 2. 버전 정책

| 항목             | 정책                                                                |
| ---------------- | ------------------------------------------------------------------- |
| 기본 버전        | 최신 **안정화(stable)** 버전 사용                                   |
| LTS 우선         | LTS 존재 시 LTS 우선 (런타임·플랫폼)                                |
| RC/beta/alpha    | **프로덕션 금지** (사내 라이브러리 등 명시적 예외만 허용)           |
| 버전 범위        | 재현 가능 빌드 보장 — 와일드카드(`*`) 금지, 범위 지정자는 신중 사용 |
| 도구 버전 고정   | 패키지 매니저·런타임 버전을 매니페스트에 명시 고정                  |
| 정기 의존성 감사 | 취약점 감사 도구를 **주기적**으로 실행 (CI 통합 권장)               |

> 모노레포는 중복 의존성을 단일 버전으로 통일(override)하여 버전 충돌을 차단한다.

## 3. 코드 품질 공통 기준

언어·프레임워크에 무관하게 아래 게이트를 **모든 프로젝트에 구성**한다.

| 게이트              | 요구사항                                                              |
| ------------------- | --------------------------------------------------------------------- |
| **pre-commit hook** | 커밋 직전 lint/format을 스테이징 파일에 자동 적용 (실패 시 커밋 차단) |
| **commit-msg hook** | 커밋 메시지 컨벤션 검증 (`COMMIT_CONVENTION.md`), 위반 시 거부        |
| **CI lint**         | PR/푸시 시 lint 무오류 필수 (머지 게이트)                             |
| **CI test**         | PR/푸시 시 자동 테스트 통과 필수 (실패 시 머지 차단)                  |
| **포매터 단일화**   | 포매팅 도구·규칙을 하나로 통일하여 diff 노이즈 제거                   |
| **에디터 일관성**   | `.editorconfig` 등으로 들여쓰기·개행·인코딩 통일                      |

**코드 리뷰 기준 (공통)**

- 모든 변경은 **리뷰 후 머지** — main/develop 직접 푸시 금지
- 리뷰 관점: 아키텍처·보안·성능·스타일 (세부는 `DEVELOPMENT.md` / `/forge:code-review`)
- 자동 검증(lint/test)을 통과한 변경만 리뷰 대상으로 인정 (증명 없는 완료 금지)

> 구체적 도구(lint·format·hook·CI 설정)는 언어별 세부 파일(§7) 참조.

## 4. 로깅 전략 (공통 원칙)

언어·프레임워크 무관 공통 원칙. **구현 코드·라이브러리 선택은 언어별 가이드로 위임**한다.

| 원칙                 | 내용                                                                             |
| -------------------- | -------------------------------------------------------------------------------- |
| **구조화 로깅 필수** | 평문 문자열 로그 금지. 키-값 구조 로그(JSON 등)로 기계 파싱 가능하게 출력        |
| **로그 레벨 정의**   | Trace/Debug / Info / Warning / Error / Fatal(Critical) 일관 적용                 |
| **Correlation ID**   | 모든 로그에 요청 추적 ID 포함 — 분산 추적·디버깅 핵심. 요청 진입점에서 발급·전파 |
| **민감 정보 제외**   | 비밀번호·토큰·개인정보·인증 헤더는 로그 출력 **금지** (마스킹·redact 필수)       |
| **레벨별 출력 분리** | 환경(dev/prod)에 따라 최소 레벨·Sink(콘솔·파일·수집기) 조정                      |

> 언어별 로깅 라이브러리·설정 예시(구조화 로거, Correlation ID 미들웨어 등)는 §7의 해당 파일 참조.

---

## 5. 디렉토리 구조 원칙

구체적 폴더 템플릿은 언어·프레임워크별로 다르므로 §7로 위임한다. 아래는 **모든 구조에 공통 적용되는 설계 원칙**이다.

| 원칙                 | 내용                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------- |
| **계층 분리**        | 프레젠테이션(UI/API) ↔ 비즈니스(도메인/서비스) ↔ 데이터(저장소/영속성) 계층 명확 분리 |
| **의존 방향 일관성** | 도메인(Core)은 외부를 참조하지 않는 순수 계층, 바깥 계층이 안쪽을 향해 의존           |
| **관심사 분리(SoC)** | 기능·역할 단위로 응집, 한 모듈이 여러 책임을 갖지 않도록 분리                         |
| **공유 코드 격리**   | 여러 앱/모듈이 쓰는 코드는 공유 패키지·공통 모듈로 분리 (모노레포는 별도 패키지)      |
| **파일명 규칙 통일** | 케이스(kebab/camel/Pascal)·접미사(`.service`, `.test` 등)를 프로젝트 내 일관 적용     |
| **설정 루트 집중**   | 빌드·lint·포맷·패키지 설정은 루트(또는 공유 설정 패키지)에 집중하여 중복 제거         |

> 언어별 구체 디렉토리 템플릿(앱 진입점·레이어 폴더·설정 파일 배치)은 §7의 해당 파일 참조.

## 6. 보안 공통 기준

| 항목                     | 정책                                                                                   |
| ------------------------ | -------------------------------------------------------------------------------------- |
| **비밀값 환경변수 관리** | 비밀번호·API 키·토큰·연결 문자열은 코드에 하드코딩 금지, 환경변수/시크릿 매니저로 주입 |
| **.env 커밋 금지**       | `.env`·자격증명 파일은 `.gitignore`에 등록, `.env.example`만 커밋                      |
| **환경변수 스키마 검증** | 런타임 진입 시 필수 환경변수 존재·형식을 검증 (누락 시 기동 실패)                      |
| **의존성 취약점 감사**   | 의존성 취약점 스캔을 정기·CI 실행, 알려진 취약 패키지 업데이트                         |
| **최소 권한 원칙**       | DB·외부 서비스 계정은 필요한 범위 권한만 부여                                          |
| **민감 정보 로그 제외**  | §4 로깅 원칙과 연계 — 비밀값·개인정보 로그/에러 응답 노출 금지                         |

## 8. 셋업 완료 검증 체크리스트 (공통)

신규 프로젝트 생성 직후 아래를 **실제 실행**하여 동작을 증명한다 (`AI_BEHAVIOR.md`: 증명 없는 완료 금지). 언어별 세부 검증 항목은 §7의 해당 파일 참조.

- [ ] 의존성 설치 성공 (지정한 패키지 매니저로)
- [ ] lint / 타입검사 무오류
- [ ] 자동 테스트 통과
- [ ] 의도적 잘못된 커밋 메시지로 `commit-msg` 훅이 **거부**되는지 확인
- [ ] 스테이징 후 `pre-commit` 훅이 lint/format을 실행하는지 확인
- [ ] 구조화 로그에 Correlation ID·레벨·구조화 필드가 출력되는지 확인
- [ ] 비밀값이 코드·로그에 노출되지 않는지 확인 (`.env` 미커밋, 환경변수 검증 동작)
- [ ] (모노레포) 빌드 시 태스크 캐시·빌드 의존 순서 동작 확인
